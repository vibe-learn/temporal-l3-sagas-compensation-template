        # temporal — Саги и компенсация

        Homework-шаблон для урока **l3_sagas_compensation** (Саги и компенсация) на платформе Vibe Learn.

        ## Что делать

        Реализуй BookingWorkflow (сага) на Go SDK (go.temporal.io/sdk): активности BookFlight,
BookHotel, BookCar и их компенсации CancelFlight, CancelHotel. Накапливай компенсации
в слайсе []func() и при сбое любого шага прогоняй их в ОБРАТНОМ порядке. Сделай BookCar
флакающим (падает по флагу), а компенсации — идемпотентными активностями с ретраями.
Тесты через TestWorkflowEnvironment с mock-активностями проверят: при сбое BookCar
вызываются CancelHotel и CancelFlight именно в обратном порядке, а при полном успехе
компенсации не вызываются.

## Контекст (из transfer-задачи урока)

Интернет-магазин, оформление заказа: (1) списать деньги с карты, (2) зарезервировать
товар на складе, (3) создать заявку на доставку у внешнего курьера, (4) отправить
клиенту письмо «заказ принят». Шаг 3 (курьерский API) периодически возвращает ошибку
«нет машин в районе». Сейчас при сбое доставки команда просто логирует ошибку — деньги
списаны, товар зарезервирован, но заказ не оформлен; клиенты пишут в поддержку.

**Вопрос:** спроектируй это как сагу в Temporal. Опиши:
(a) компенсацию для каждого из первых трёх шагов и порядок отката при сбое шага 3;
(b) почему компенсации должны быть идемпотентными активностями с ретраями;
(c) где здесь pivot и куда поставить отправку письма.

## Recap из урока

- **Saga** — альтернатива 2PC для кросс-сервисных процессов: вместо атомарного отката у каждого шага своя **компенсация**, при сбое — откат уже сделанных шагов в **обратном порядке**.
- Saga даёт **eventual consistency**, не атомарную: нет блокировок, промежуточное состояние видно, но в итоге система приходит к «всё сделано» или «всё откачено».
- Компенсация — **семантический откат** (refund, отмена брони, письмо-извинение), а не магический rollback состояния. Реальный мир не откатывается — ты компенсируешь его эффект.
- Компенсации сами должны быть **идемпотентными активностями с ретраями**: при падении воркера откат может повториться, а отмена тоже дёргает внешний сервис.
- Ставь компенсируемые шаги ДО **pivot** (точки невозврата), необратимые — ПОСЛЕ. Durable workflow не даст откату застрять — дожмёт компенсацию через падения воркеров.

        ## Как работать

        1. Платформа Vibe Learn создаёт копию этого репо в твоём GitHub-аккаунте по клику «Начать домашку» на странице урока (через GitHub `/generate`, codecrafters-pattern).
        2. Склонируй копию локально, реализуй TODO в `main.go` (workflow + активности), прогони тесты, запушь.
        3. CI (`.github/workflows/ci.yml`) запускает `go vet` + `go test ./...` на каждый push. Платформа слушает результат через webhook от GitHub Actions и обновляет статус домашки на странице урока.

        ## Локальное окружение

        - Go 1.22+
        - SDK: `go.temporal.io/sdk`
        - Docker + docker-compose — `docker compose up` поднимает Temporal dev server на `:7233` + Web UI на `:8233`. Адрес переопределяется через env `TEMPORAL_ADDRESS` (дефолт `localhost:7233`).
        - Юнит-тесты на `testsuite.TestWorkflowEnvironment` (активности замоканы) бегут в CI БЕЗ сервера; интеграционный тест включается через `TEMPORAL_INTEGRATION=1`.

        ## Запуск

        ```bash
        # Поднять локальный Temporal dev server + UI
        docker compose up -d
        # Web UI: http://localhost:8233

        # Прогнать тесты (юнит на TestWorkflowEnvironment — без сервера;
        # интеграционный включается через TEMPORAL_INTEGRATION=1)
        go test ./...
        TEMPORAL_INTEGRATION=1 go test ./...

        # Запустить воркер (регистрирует workflow + активности, слушает task queue)
        go run .
        ```
